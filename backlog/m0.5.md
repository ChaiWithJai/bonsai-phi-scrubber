# M0.5 — Feasibility spike (GATES M1) ⚡
*Goal: verify the three load-bearing empirical claims the business case rests on — cheaply, on hardware we already have, BEFORE building the Rust core around assumptions. Throwaway code (`spike/`), not the real core. This IS "earn the map" (Constitution III).*
*Depends on: a handful of golden notes (authored here, fold into M0 after). Output: a go/no-go report `spike/REPORT.md`.*

## Why this exists
The backlog tells the build order; it does not tell which assumptions are unproven. Three are, and if any fails the design changes:
1. **Recall** — can `rules ∪ Bonsai-1.7B` hit **≥0.99 recall / ~0 leakage**, especially on the **hard case** (a name buried in prose)? A ternary 1.7B model that can't freehand JSON is being asked to catch contextual PHI.
2. **R1 / on-device** — does 1.7B *text* even run acceptably on iPhone-11-class (A13) hardware? Only the 4B *image* model on a 2023 phone is locally proven.
3. **Determinism** — is ternary Bonsai byte-stable at temp-0 greedy, enough to commit a matching `golden-run.txt`?
