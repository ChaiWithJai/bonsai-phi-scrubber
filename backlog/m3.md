*Goal: the iOS shell runs the loop on-device, offline. Depends on: M1 ✅ (NOT M2 — device proof rides the core, not the desktop loop).*

**This milestone opens with a measurement, not an assumption.** The headline claim is "iPhone 11, 2019." The only locally-proven on-device Bonsai run is a *different* model (4B image) on a *2023* phone (iPhone 17 Pro Max). See risk R1 in the design spec and `docs/seed/bonsai-ecosystem-brief.md`.

First task is the gate; everything else is blocked on it:
- [ ] **M3-T00** *(gate)* — Stand up mlx-swift loading the Ternary-Bonsai-1.7B **MLX** text weights; measure tokens/sec and memory on the **oldest device we can get** (ideally A13/iPhone 11). **STOP and report to Jai with the number.** Decide: keep "2019," soften the phrasing, or let Tier-1 (CLI) carry reproduction. Do not proceed to T01+ until Jai rules.
- [ ] **M3-T01** — iOS `InferenceProvider` adapter: in-process mlx-swift; **client-side** constrained decoding (GBNF or validate-and-retry — no `response_format` server on device).
- [ ] **M3-T02** — iOS `SecureStore` (Secure Enclave / Keychain): raw input + redaction map, never leave hardware.
- [ ] **M3-T03** — `Capture` (on-device ASR + text) and the airplane-mode UI: consent → toggle → dictate → Scrub → structured card → Send blocks while offline → flush on reconnect.
- [ ] **M3-T04** — UniFFI binding of `airplane-core` → Swift; prove on-device eval parity with the CLI on the golden notes.

**Done-when:** the airplane-mode loop runs offline on a real device, on-device eval matches the CLI, and Jai has accepted the device story. → HITL.
